# JD 自检 · 自动化匹配

当用户请求对一份新意向岗位/JD 做自检，或首次使用需建立求职画像时，按本 Skill 执行。

## 触发场景

**场景零：首次使用 / 建立求职画像**

满足其一即进入场景零（简历 + 访谈 → 产出/完善「工作需求-匹配自检」）：(1) **显式**：用户说「帮我建立求职画像」「初始化求职自检」「第一次用、还没有自检文档」等；(2) **发简历**：用户直接发来简历（粘贴或附文件），可视为要建立/更新求职画像；(3) **表达求职意向**：用户说「我想找工作」「准备找工作」「在找机会」等；(4) **隐式**：用户请求做 JD 自检（如「对这份 JD 做自检」并提供 JD），但工作区内不存在或为空的自检文档时，先引导进入场景零，完成后再继续自检。

**场景零执行步骤**：1. 请用户提供简历（粘贴或给出路径），并说明当前求职方向/目标岗位类型；若用户仅有 PDF/Word，可请其粘贴关键内容。2. 按「工作需求-匹配自检」文档所需结构（MBTI、经验能力、价值观、特长、技能概览、硬性约束、匹配自检清单）进行**访谈**：逐块或分组提问，至少覆盖经验与能力、求职方向与偏好、硬性约束（地点/薪资/节奏）。3. 问完所需问题后，**先向用户呈现求职画像摘要**（或关键填写结果），**请用户确认**是否按此生成/写入「工作需求-匹配自检」；**仅在用户确认后**再创建或更新该文档的第二节～第八节；若用户已有该文档但缺项，仅补充缺失部分。

---

**场景一：新 JD 自检**

用户**发了一个新的 JD，并说明这是新的意向岗位**时，应用本 Skill。例如用户说：

- 「这是新的意向岗位」+ 粘贴或附上 JD
- 「给你发了一个新的 JD，这是新的意向岗位」

也包含用户明确请求自检时的说法，如：「对这份 JD 做自检」「新 JD 自检」「根据我的技能点匹配这份岗位」「帮我对这个岗位做自检」等，并提供了 JD 或工作考虑文档。

**场景二：用户准备投递某岗位**

用户说「我想投递 XX」「准备投 XX」「打算投 XX」等，表明用户准备投递某岗位时，执行「投递准备流程」（见下文步骤 6.5），为用户生成 Boss 直聘打招呼语。

**场景三：用户告知已投递某岗位**

用户说「我已经投递了 XX」「XX 已投递」「刚投了 XX」等，表明某岗位状态需从「待投递」变为「已投递」时，执行「投递状态更新流程」（见下文步骤 7）。

**场景四：用户告知获得面试邀请**

用户说「XX 约面试了」「收到 XX 的面试邀请」「XX 要面试了」等，表明某岗位进入面试阶段时，执行「面试邀请处理流程」（见下文步骤 8）。

## 前置条件

使用 JD 自检（场景一及后续步骤）前，需已完成「简历 + 访谈」并存在有效的「工作需求-匹配自检」文档；若未完成，先执行**场景零**。

## 数据来源

- **唯一画像与清单来源**：工作区中的 `工作需求-匹配自检.md`。必须读取该文档的第二节～第七节（MBTI、经验能力、价值观、特长、技能概览、硬性约束）以及第八节（匹配自检清单）。不引入其他文档作为技能点依据。
- **基础简历模板**：`lufi-简历.md`。生成定制简历时以此为结构与内容基础，根据 JD 调整「个人优势」顺序和措辞，工作/项目经历保持框架不变但可调整强调点。
- **投递看板**：`岗位投递看板.md`（Obsidian Kanban 格式）。用于追踪岗位投递状态，包含「待投递」「已投递」「面试中」「结果出」四列。

## 配置说明

以下路径与命名需根据用户本地环境配置：**自检文档路径**（如 `工作需求-匹配自检.md`）、**简历模板路径**（如 `lufi-简历.md`）、**看板路径**（如 `岗位投递看板.md`）、**工作考虑文档的目录与命名约定**（如 `{公司}/{公司} {岗位} 工作考虑.md`）。使用前请将 SKILL 中出现的上述路径改为你本地的文档路径。

## JD 输入方式

- **方式一**：用户在对话中直接粘贴 JD 全文 → 对该文本做自检。
- **方式二**：用户给出某份「工作考虑」文档路径（如 `富途/富途 交易体验产品经理 工作考虑.md`）→ 从该文档中提取「岗位职责」「岗位要求」「公司 & 岗位基本信息」等作为 JD 输入。

## 执行流程（顺序不可颠倒）

### 1. 比对与待补充判断

- 用自检文档中的技能点与「八、匹配自检清单」逐条对照 JD。
- 若 JD 要求的能力、经历或证书在自检文档中**未写清或无法判断用户是否具备**（例如「是否有某类交易经验」「某证书是否已考」），**不得猜测**；须明确列出「待你补充」的项，并向用户提问，**不要在此阶段输出最终结论**。

### 2. 若有「待补充」项

- 仅输出「待你补充的信息」列表并逐项提问，请用户补充。
- 用户补充后：
  1. **先**将用户补充的技能/经历/证书等内容同步更新到 `工作需求-匹配自检.md` 的对应小节（如「四、工作经验与核心能力」「六、特长与差异化优势」「七、目前技能概览」等）；若某条不知归入哪一节，写入最相关的一节并注明「（JD 自检时补充）」。
  2. **再**进入步骤 3，在最后统一输出完整结论。

### 3. 结论在最后输出

仅当以下之一满足时才输出结构化结论：

- 无待补充项；或  
- 待补充项已由用户补充，且已写回 `工作需求-匹配自检.md`。

**结论输出结构**（按顺序）：

1. **逐条自检结果**：按「八、匹配自检清单」顺序，每条给出「符合 / 存疑 / 不符合」及一句理由。
2. **硬性约束**：地点、弹性、薪资、节奏在 JD 中的可判断程度，以及是否需面试时问清。
3. **综合匹配度**：星级或高/中/低 + 2～3 句小结。
4. **是否建议投递**：建议投 / 可投但需先问清 XX / 建议慎重或暂不投，及理由。
5. **沟通时必问项**：从自检文档「硬性约束」与「待确认」逻辑推导出的 3～5 个问题。

（若有待补充项，仅在用户补充并写回之前列出「待你补充的信息」；最终结论中不再含待补充项，而是已纳入更新后的匹配结果。）

### 4. 追问是否写回工作考虑文档

在结论输出后，**必须**追问一句：是否将本次结论写入某份工作考虑文档？

- 若用户指定文档路径，则将「契合点摘要」「综合评估」「是否值得投」「必问项」写入或追加到该文档的对应小节（如「和我的契合点」「综合评估」「待确认」等）；若文档尚无对应结构则追加。

### 4.5 若值得投，追问是否添加到投递看板

**仅当**综合评估为「值得投」「建议投」或「可投」时，在步骤 4 完成后追问：是否将此岗位添加到投递看板？

若用户确认，在 `岗位投递看板.md` 的「待投递」列末尾添加一行：

```markdown
- [ ] [[{公司}/{公司} {岗位} 工作考虑|{公司} {岗位}]]
```

**注意**：
- 添加前先读取看板文件，检查是否已存在该岗位（避免重复添加）。
- 若已存在，告知用户「该岗位已在看板中」，不重复添加。
- 后续状态更新（已投递、面试中、结果出）由用户在 Obsidian 看板视图中拖拽完成，无需通过本 Skill。

### 5. 若值得投，追问是否生成定制简历

**仅当**综合评估为「值得投」「建议投」或「可投」时，在步骤 4 完成后追问：是否为该岗位生成定制简历？

若用户确认，按以下逻辑生成：

1. **读取基础模板**：`lufi-简历.md`。
2. **调整「个人优势」**：
   - 根据 JD 要求重新排序，将与该岗位最匹配的能力放在前面。
   - 措辞调整：突出 JD 关键词对应的经验与能力，删除或弱化与该岗位无关的内容。
3. **调整工作/项目经历**：
   - 结构与条目保持不变。
   - 根据 JD 调整强调点和措辞（如某段经历与 JD 高度相关，可加粗或扩展描述）。
4. **更新头部信息**：
   - 求职意向：改为该岗位名称。
   - 期望城市：改为 JD 标注的城市。
   - 期望薪资：保持或根据 JD 调整。
5. **末尾注明**：`*本简历针对「{公司} {岗位名称}」岗位定制。*`
6. **输出位置**：在该岗位对应的文件夹中创建，命名为 `{岗位名称}_简历.md` 或你在配置中约定的格式（如 `{公司}/{岗位名称}_简历.md`）。

### 6. 简历确认后导出 PDF

生成定制简历后，若用户确认简历内容没有问题（或经过修改后确认），追问：是否导出 PDF？

若用户确认，按以下方案导出：

**方案 A：Chrome Headless 自动导出**（推荐，全自动）

1. 读取简历 `.md` 文件内容。
2. 生成带样式的 HTML 文件，保存在同目录下，文件名与 `.md` 文件相同，扩展名为 `.html`。
3. **HTML 样式要求**（黑白简洁风格，一页 A4）：

```css
@page { size: A4; margin: 12mm 15mm; }
* { margin: 0; padding: 0; box-sizing: border-box; }
body {
  font-family: -apple-system, BlinkMacSystemFont, "Segoe UI", Helvetica, Arial, sans-serif;
  font-size: 11px;
  line-height: 1.5;
  color: #1a1a1a;
  background-color: #fff;
  max-width: 210mm;
  padding: 20px;
}
h1 { font-size: 22px; font-weight: 600; color: #000; margin-bottom: 8px; }
.header-info { font-size: 11px; color: #1a1a1a; margin-bottom: 2px; }
.header-summary { font-size: 11px; color: #555; margin-bottom: 16px; }
h2 { font-size: 15px; font-weight: 600; color: #000; margin-top: 18px; margin-bottom: 10px; }
h3 { font-size: 11.5px; font-weight: 600; color: #1a1a1a; margin-top: 12px; margin-bottom: 4px; }
ul { list-style: disc; padding-left: 20px; margin-bottom: 8px; color: #1a1a1a; }
li { margin-bottom: 4px; }
li::marker { color: #888; }
strong { font-weight: 600; color: #000; }
```

4. HTML 结构：
   - `<h1>` 姓名
   - `<p class="header-info">` 基本信息（性别、电话、邮箱）
   - `<p class="header-summary">` 学历、经验摘要、求职意向、薪资、城市
   - `<h2>` 大节标题（个人优势、工作经历、项目经历、教育经历）
   - `<h3>` 公司/项目名称
   - `<ul><li>` 列表项，加粗部分用 `<strong>`
5. 使用 Chrome Headless 自动导出 PDF：
   ```bash
   "/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --headless --disable-gpu --print-to-pdf="{PDF文件路径}" --no-margins "{HTML文件路径}"
   ```
6. 导出成功后，删除 HTML 文件并打开 PDF 供用户确认：
   ```bash
   rm "{HTML文件路径}"
   open "{PDF文件路径}"
   ```

**方案 B：浏览器手动打印**（备选，Chrome 不可用时）

1. 完成方案 A 的步骤 1-4。
2. 使用 `open "{HTML文件路径}"` 在浏览器中打开。
3. 提示用户：按 `Cmd + P` → 目的地选「另存为 PDF」→ 边距选「最小」→ 保存。

### 6.5 生成 Boss 直聘打招呼语（场景二触发 / PDF 导出后自动触发）

以下情况触发本步骤：
- **PDF 导出成功后**：自动为用户生成打招呼语。
- **用户说「我想投递 XX」**（场景二）：读取该岗位的「工作考虑」文档获取 JD，结合用户画像生成打招呼语。

**生成要求**：

1. **字数控制**：100～200 字（Boss 直聘打招呼有字数限制）。
2. **结构**：
   - 开头：简短自我介绍（1 句）
   - 核心：与该岗位最匹配的 2～3 个优势/经历（2～3 句）
   - 结尾：表达兴趣与期待（1 句）
3. **风格**：真诚、简洁、专业，不套话、不空泛，突出与 JD 的具体匹配点。
4. **输出格式**：直接输出打招呼语文本，方便用户复制。

**示例结构**：

```
您好，我是 {姓名}，{学历/经验概述}。

看到贵司 {岗位名称} 的招聘，非常感兴趣。我在 {相关经历 1}，{相关经历 2}，与岗位要求的 {JD 关键词} 高度契合。

期待有机会进一步交流，谢谢！
```

### 7. 投递状态更新流程（场景三触发）

当用户告知「我已经投递了 XX 岗位」时，执行以下步骤：

1. **识别岗位**：从用户消息中提取公司名称和岗位名称。
2. **读取看板**：读取 `岗位投递看板.md`。
3. **查找并移动**：
   - 在「待投递」列中查找该岗位对应的行（匹配公司名或岗位名）。
   - 若找到，将该行从「待投递」列删除，添加到「已投递」列末尾。
   - 若未找到（可能已在「已投递」或其他列），告知用户当前状态，不重复操作。
4. **确认更新**：告知用户「已将 {公司} {岗位} 移动到『已投递』列」。

### 8. 面试邀请处理流程（场景四触发）

当用户告知「XX 约面试了」「收到 XX 的面试邀请」时，执行以下步骤：

1. **识别岗位**：从用户消息中提取公司名称和岗位名称。
2. **读取看板**：读取 `岗位投递看板.md`。
3. **查找并移动**：
   - 在「待投递」或「已投递」列中查找该岗位对应的行。
   - 若找到，将该行移动到「面试中」列末尾。
   - 若已在「面试中」列，告知用户「该岗位已在『面试中』列」。
4. **确认更新**：告知用户「已将 {公司} {岗位} 移动到『面试中』列」。
5. **询问面试准备**：追问「是否需要我帮你做面试准备？」
   - 若用户确认，读取该岗位的「工作考虑」文档，结合 JD 要求和用户画像，输出：
     - **岗位核心要求回顾**：JD 中的关键能力与职责
     - **你的匹配点**：可重点展示的经历与能力
     - **可能被问到的问题**：基于 JD 推测的 5～8 个面试问题
     - **建议准备的回答要点**：每个问题的回答思路或 STAR 结构建议
   - 若用户不需要，流程结束。

## 规则摘要

- **不猜测**：无法从自检文档判断用户是否具备某技能/经历时，列出待补充项并提问。
- **结论放最后**：有待补充时，先完成提问 → 用户补充 → 写回自检文档中的用户信息 → 再输出完整结论。
- **补充必写回**：用户补充的信息必须同步更新到 `工作需求-匹配自检.md`，便于后续 JD 自检沿用。
- **简历仅值得投时生成**：综合评估为「值得投」「建议投」「可投」时才追问是否生成定制简历；若评估为「慎重」「暂不投」则跳过此步骤。
- **PDF 在简历确认后导出**：简历生成后，待用户确认内容无误（或修改完成），再追问是否导出 PDF；优先使用 Chrome Headless 自动导出，若 Chrome 不可用则回退到浏览器手动打印。
- **PDF 导出后生成打招呼语**：PDF 导出成功后，自动生成一段 Boss 直聘打招呼语（100～200 字），突出与 JD 匹配的核心优势，方便用户直接复制使用。
- **状态更新与面试准备**：用户告知已投递时，更新看板到「已投递」；用户告知获得面试邀请时，更新看板到「面试中」并询问是否需要面试准备。
